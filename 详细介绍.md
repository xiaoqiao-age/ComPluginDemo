# ComPluginDemo 项目详细介绍

## 一、项目定位

一个 **免注册 COM (Reg-Free COM)** 插件系统演示项目，使用 Windows SxS (Side-by-Side) 清单替代注册表，实现跨语言 COM 组件的加载和协作。无需 `regsvr32` / `regasm`，部署即用。

---

## 二、技术栈

| 技术 | 版本 | 用途 |
|------|------|------|
| .NET 8 | net8.0 | COM 宿主和 C# COM 组件运行时 |
| Avalonia | 11.2.3 | 跨平台 UI 框架 (本项目用于 Windows) |
| CommunityToolkit.Mvvm | 8.4.0 | MVVM 模式支持 |
| VB6 | 6.0 | VB6 ActiveX DLL COM 组件 |
| Windows SxS | — | 免注册 COM 的清单加载机制 |
| Win32 ActivationContext API | — | 动态加载 COM 清单 |

## 五、两组协作关系

### 关系 1 (Calculator → VB6 StringProcessor)

```
A 加载 Calculator   → 得到 Calculator COM 对象
A 加载 VB6 StringProcessor
A 调用 vb6Obj.SetCalculatorObject(calculator.ComObject)
  → VB6 保存引用，后续用 Calculator 算数学表达式
```

### 关系 2 (ServiceC → OrchestratorB)

```
A 加载 ServiceC     → 得到 ServiceC COM 对象
A 加载 OrchestratorB
A 调用 orchestratorObj.SetServiceCObject(serviceC.ComObject)
  → B 保存引用，后续用 ServiceC 做编码/哈希
```

两组关系模式完全相同：**A 做桥梁，把工具组件传给编排组件**。

## 六、各插件功能详情

### 6.1 Calculator (数学计算器)

递归下降解析器，支持四则运算。

| 输入 | 输出 |
|------|------|
| `3+4*(2-1)` | `7` |
| `(10+5)/3` | `5` |
| `100/0` | `错误: 除数不能为零` |

### 6.2 VB6 StringProcessor (字符串处理器)

VB6 ActiveX DLL，接收 Calculator 对象后可以调用 C# 计算器功能。

### 6.3 ServiceC (编码/哈希工具服务)

| 命令 | 示例 | 说明 |
|------|------|------|
| `base64enc <文本>` | `base64enc Hello` → `SGVsbG8=` | Base64 编码 |
| `base64dec <文本>` | `base64dec SGVsbG8=` → `Hello` | Base64 解码 |
| `urlenc <文本>` | `urlenc a b&c` → `a%20b%26c` | URL 编码 |
| `urldec <文本>` | `urldec a%20b%26c` → `a b&c` | URL 解码 |
| `sha256 <文本>` | `sha256 Hello` → `185f8db3...` | SHA256 哈希 |
| `md5 <文本>` | `md5 Hello` → `8b1a9953...` | MD5 哈希 |

### 6.4 OrchestratorB (编排器)

B 自己不做编码/哈希，全部委托 ServiceC 执行，价值在于**组合编排**。

| 命令 | 示例 | 说明 |
|------|------|------|
| `encode-hash <文本>` | `encode-hash Hello` | 先 Base64 编码再 SHA256 哈希，返回两步结果 |
| `direct <C命令> <参数>` | `direct base64enc Hello` | 直接转发给 ServiceC 执行 |
| `pipeline <命令链> <文本>` | `pipeline base64enc\|sha256 Hello` | 管道式串联多个 ServiceC 命令 |
| `info` | `info` | 显示 ServiceC 连接状态 |

**`encode-hash` 执行流程示例：**

```
用户输入: encode-hash Hello
  → B 调用 C.Execute("base64enc Hello")   → "SGVsbG8="
  → B 调用 C.Execute("sha256 SGVsbG8=")   → "4dc72105..."
  → 返回:
    Base64: SGVsbG8=
    SHA256: 4dc72105...
```

**`pipeline` 执行流程示例：**

```
用户输入: pipeline base64enc|sha256 Hello
  初始值: Hello
  [base64enc] → SGVsbG8=
  [sha256]    → 4dc72105...
  最终结果: 4dc72105...
```

## 七、核心机制详解

### 7.1 免注册 COM 加载 (两种策略)

```
策略 1 (优先): SxS 清单加载
  plugins.json → ManifestPath → ActivationContext API
  → CreateActCtx() → ActivateActCtx() → GetTypeFromProgID()
  → Activator.CreateInstance() → Initialize()
  结果: LoadSource = "Manifest"

策略 2 (回退): 注册表加载
  → Type.GetTypeFromProgID(ProgId)
  → Activator.CreateInstance() → Initialize()
  结果: LoadSource = "注册表" 或 "注册表 (manifest失败回退)"
```

回退策略的意义：VB6 IDE 调试时，组件已注册到注册表但 DLL 不在部署目录，删掉 manifest 文件即可走注册表路径进行调试。

### 7.2 SxS 清单搜索顺序

```
宿主 app.manifest 声明: <dependency> name="CSharpComPlugin"
  → SxS 搜索:
    1. exe 同目录: CSharpComPlugin.manifest
    2. 同名子目录: CSharpComPlugin\CSharpComPlugin.manifest  ← 采用这种

组件 manifest 声明依赖 (嵌套):
  OrchestratorBPlugin.manifest 声明 <dependency> ServiceCPlugin
  → SxS 在 OrchestratorBPlugin\ 下搜索:
    OrchestratorBPlugin\ServiceCPlugin\ServiceCPlugin.manifest
```

### 7.3 组件间通信 — 纯 COM 晚绑定

```csharp
// A 把 C 传给 B (MainWindowViewModel.cs)
dynamic orchestratorObj = orchestratorB.ComObject!;
orchestratorObj.SetServiceCObject(serviceC.ComObject);  // IDispatch 调用

// B 内部使用 C (OrchestratorBPlugin.cs)
dynamic serviceC = GetServiceC();                       // 取回 C 的引用
string result = serviceC.Execute("base64enc Hello");    // IDispatch 调用
```

所有跨组件调用通过 `dynamic` + COM IDispatch 机制，**没有** .NET ProjectReference，完全解耦。

### 7.4 B 获取 C 的双路径

| 路径 | 方式 | 代码 | 优先级 |
|------|------|------|--------|
| A 传递 | `SetServiceCObject(obj)` | 存入 `_serviceCObj` | 优先使用 |
| B 独立创建 | `Type.GetTypeFromProgID("ComPluginDemo.ServiceC")` | 存入 `_ownServiceC` | 备用 |

```csharp
// B 内部选择逻辑
private dynamic? GetServiceC()
{
    return _serviceCObj ?? _ownServiceC;  // 优先宿主传递，其次独立创建
}
```

B 独立创建 C 依赖 `OrchestratorBPlugin.manifest` 中声明的 `<dependency> ServiceCPlugin`，SxS 在 `OrchestratorBPlugin\ServiceCPlugin\` 目录下找到 C 的清单和 DLL。

### 7.5 插件生命周期

```
加载:   CreateActCtx → ActivateActCtx → GetTypeFromProgID → CreateInstance → Initialize()
执行:   dynamic 调用 Execute(input)
传递:   dynamic 调用 SetXxxObject(comObj)    ← 宿主桥梁
卸载:   Shutdown() → Marshal.ReleaseComObject() → DeactivateActCtx → ReleaseActCtx
```

## 八、COM 接口与 GUID 清单

### 8.1 接口定义

每个 C# COM 组件定义自己的接口，DispId 1-6 签名一致：

```csharp
[DispId(1)] string Name { get; }
[DispId(2)] string Version { get; }
[DispId(3)] string Description { get; }
[DispId(4)] void Initialize();
[DispId(5)] string Execute(string input);
[DispId(6)] void Shutdown();
```

OrchestratorB 额外定义 DispId 7：

```csharp
[DispId(7)] void SetServiceCObject(object serviceCObj);
```

## 十、关键设计决策

| 决策 | 理由 |
|------|------|
| **全部 x86** | VB6 只有 32 位，宿主和所有 COM 组件必须匹配 |
| **InterfaceIsDual** | 同时支持 vtable 早绑定 (C#) 和 IDispatch 晚绑定 (VB6) |
| **dynamic 晚绑定通信** | 跨组件调用不需要编译期引用，保持 COM 隔离 |
| **B 不 ProjectReference C** | 保持纯 COM 隔离，和 VB6 → Calculator 模式一致 |
| **B 双引用 C** | 宿主传递优先 + 独立创建备用，提高健壮性 |
| **manifest 优先 + 注册表回退** | 生产用 manifest (免注册)，调试用注册表 (VB6 IDE) |
| **ContinueOnError="WarnAndContinue"** | VB6 锁文件时不阻塞其他插件的复制目标 |
| **每个组件独立 GUID** | 各组件有自己的接口 GUID 和类 GUID，互不冲突 |
| **plugins.json 顺序** | ServiceC 在 OrchestratorB 之前加载，确保传递时 C 已就绪 |
